# 交接（前端）：生成中轮询加密（20s → 3-5s，兑现后端 15s TTFC）

> 纯前端、零契约变化、零后端改动。消费已有的 `generating` 字段（2026-07-04 加法）。

## 背景：后端 15s 就绪,前端 20s 才来看一眼

后端 #257 已把懒生成首屏(请求语言 guide 落库)压到 **~15s**(staging 实测 15.6s,含 2s 轮询粒度)。
但真机日志实锤:**App 对 content 端点的轮询间隔是 ~20s**(`17:41:00 → 17:41:20 → 17:41:40`,精确 20s):

| 层 | 时间 |
|---|---|
| 服务端 zh guide 落库(全新 stub,en 也从零) | **~15s** |
| App 第 2 次轮询才看到 | **20s 或 40s** → 用户感知 "30s 左右" |

后端投资被轮询粒度吃掉 5-25s。

## 改法（一处间隔逻辑）

- `generating == true` 期间(懒生成/懒翻译进行中):轮询间隔 **3-5s**;
- `generating == false`:恢复现有慢轮询/停止(别常态高频打接口);
- 已有的诚实进度展示(`generation: {published, expected}` 分数、阶段文字)不动,只是数字更新得更勤。

⚠️ 别全局改成 3-5s——只在 `generating=true` 的窗口内加密,窗口通常 1-2 分钟,服务端扛得住(content 读接口便宜)。

## 验收（真机）

1. 挑一件**全新未生成**的藏品(列表里"待完善"状态),点开 → 主讲解应在 **~16-20s** 内出现(而非 30-40s);
2. 深度段/问答在其后 1-2 分钟内陆续填充,进度分数随轮询更新;
3. 生成完成后(`generating=false`)抓包确认轮询已回落/停止。

## 已知边缘(不修,知悉即可)

**生成中途切语言**(如 fr 触发生成后立即切 zh):新语言撞生成锁,需等当前件全部生成完(~1min)+ 懒翻译(guide 已是先翻先落,~7s 内出)→ 总感知 ~1-2min。属锁的固有串行,场景少,后端 backlog;前端无需处理,轮询加密后该路径也会在翻译落库的第一时间显示。
