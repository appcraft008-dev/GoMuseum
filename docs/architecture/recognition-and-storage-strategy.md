# 识别与存储策略（决策记录）

> 日期 2026-06-13。两次讨论的结论固化，作为后续技术选型的指导依据。
> 关联：第 0 步识别摸底 GO（`backend/tools/recognition_eval/REPORT.md`）、第 1 步数据与存储地基。

## 一、识别能力判断：博物馆 ≠ 植物，本质上更简单

第 0 步评测在真实游客照（v3，33 张经视觉闸核验）上达到 **94%（修正后 97%）作品命中、0 全错、有用率 100%**，结论 GO。

为什么藏品识别远比植物识别（参考 myPlantMate）容易——是**问题类型不同**，不是程度差异：

| | 植物识别 | 博物馆藏品识别 |
|---|---|---|
| 集合 | 开放集，约 40 万种，仍在变 | **封闭集**，单馆几千件，已知且固定 |
| 类内差异 | 极大（季节 / 部位 / 生长期 / 健康 / 个体） | **几乎为零**：每件藏品是一个固定不变的视觉实例 |
| 问题本质 | 细粒度分类 | **实例检索 / 近重复图像匹配** |

植物难在"这一株是 40 万种里的哪种，而它本身还在变"；一幅画就是一张固定的标准参考图。藏品识别因此更接近反向图搜 / 地标识别，准确率天然更高。

**唯一带"植物味"的难点：系列/版本作品**（梵高多版《阿尔的卧室》、莫奈睡莲系列、塞尚多版《玩牌者》）——近乎相同的作品需更细区分，但范围比植物小到可忽略。次要难点是拍摄条件（强反光/大斜角/遮挡/暗光/裁切），评测保留意见里的最坏场景未覆盖。

## 二、长期可自建识别，省 GPT 成本 + 提速——走「检索」而非「训练分类器」

可以像植物那样自建模型替代 GPT 调用，但**落地形态是 embedding 向量检索，不是训练分类器**：

1. 用预训练视觉编码器（CLIP / DINOv2 等，开源）把每件藏品的参考图编码成向量，建向量索引（pgvector / FAISS / Qdrant）；
2. 用户拍照同样编码 → 最近邻搜索 → 返回最相似藏品 + 相似度分数（阈值判置信）。

为什么检索优于训练分类器：

- **加新藏品不用重训**：新增 = 编码一张参考图、插一条向量。分类器每次馆藏变动都要重训，对"多博物馆"产品是致命负担。
- **又快又便宜**：一次小模型前向（CPU 可跑，几十毫秒）+ 向量检索（毫秒级），**零 GPT API 成本**。
- **准确率高**：实例匹配正是这类 embedding 的设计目的。
- 置信不足时给 top-N 候选，UX 上"给你看最像的 3 件，你点确认"，比"猜一个还可能错"更稳健。

### 演进路线（两段式，平滑过渡）

```
现在（启动期）：拍照 → GPT 识别 + 生成内容        ← 量小，API 成本可忽略，先跑通
未来（规模化）：拍照 → 本地 embedding 检索命中 → 直接返回（零 API、毫秒级）
                       └ 检索没把握 → fallback 到 GPT
```

**不要现在就建索引（过早优化）**——量小时 GPT 成本微不足道。但**地基现在就要按"未来能建索引"铺好**。

## 三、图片存储：自存 R2 vs 只存 URL

成本担忧成立，但要瞄对对象——**真正随用户规模暴涨的是用户照片，不是藏品图**：

| | 藏品参考图（识别+展示） | 用户拍的照片 |
|---|---|---|
| 数量 | **有界**：单馆几千件，每件 1 张 | **无界**：随用户量/使用次数线性暴涨 |
| 增长来源 | 馆藏规模（小、慢） | 用户规模（这才是会"越来越贵"的那块） |
| 是否长期存 | 是（产品命脉） | **基本不需要——识别完即弃** |

### 决策

1. **藏品参考图：源 URL + 自存 R2 副本，两个都存。**
   - 存 `source_url`：署名（CC 图必须标注）、溯源、将来重抓。
   - 镜像一份到 R2，对用户展示走自存副本——稳定、可控、可转 WebP/AVIF 提速，**R2 出口流量免费**。
   - 不可只存外链：Flickr 图用户可随时删除、Wikimedia 文件会改名/移动、博物馆官网会改版、大规模热链会被限流/封 UA。"URL 永远不变"不成立。
   - 实账：奥赛 3000 件 × ~1MB ≈ 3GB ≈ **$0.045/月**；整个卢浮宫 35000 件 ≈ **$0.5/月**。由馆藏规模封顶，不随用户增长。
2. **用户照片：默认不长期存**，识别完即弃或极短 TTL，从源头掐掉随用户暴涨的成本。
3. 可选中间路：懒加载镜像（首次访问再抓进 R2）。但几千张直接全镜像更简单，倾向全镜像。

## 四、对第 1 步数据地基的约束

- 每件藏品数据模型含 **`source_url`（外部源，溯源+署名）** 和 **`image_key`/`stored_url`（R2 自存副本，展示用）** 两个字段。
- 预留 **embedding 向量字段/向量表**，让未来检索索引能无痛接上。
- 数据模型从一开始 **泛化到「多博物馆 × 多藏品」**，不写死奥赛。
- 用户照片不进库。
