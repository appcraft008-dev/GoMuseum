# GoMuseum 数据与内容架构设计

> 状态：草案 v1（2026-06-15）｜作者：hongyang + Claude
> 关联：[recognition-and-storage-strategy.md](./recognition-and-storage-strategy.md)、[../PRE_PAID_ROADMAP.md](../PRE_PAID_ROADMAP.md)

本文回答一个核心问题：**当后端需要庞大的博物馆藏品数据（每馆数万至数十万件，含硬数据、多种描述、解说语音）时，如何在 staging 充分测试后再上 prod，而不必维护两套越来越庞大的数据？**

---

## 1. 核心原则：环境隔离的是「代码 + schema」，不是「数据」

把"数据"和"环境"绑在一起想，会得到一个假两难：要么在 staging 维护一份全量数据（两套庞大副本），要么 staging 没数据无法测试。

真正的模型是：

| | staging | prod |
|---|---|---|
| 数据量 | **小样本**（覆盖所有类型/边界/语言的代表性子集） | **全量** |
| 来源 | 跑富化管线灌样本 | 跑**同一套已验证的管线**灌全量 |
| 关系 | **不互相拷贝数据**。提升的是「管线（代码）」，在各自环境各跑各的 |

**推论：不存在"两套庞大数据"。staging 永远是小样本；prod 全量但只有一份；数据从不在环境间复制。**

这成立的前提是 **数据可重现**（见 §2）。GoMuseum 当前的藏品硬数据是纯 Wikidata 抓取，天然满足。

---

## 2. 数据三分类与各自归位

| 类别 | 例子 | 来源 | 性质 | 存储 |
|---|---|---|---|---|
| **硬 facts** | 标题、作者、年代、流派、尺寸、馆藏号、图片 URL | Wikidata | 免费、确定、**可随意重抓** | PostgreSQL（结构化字段 + 原始 JSONB 整包） |
| **叙事 narrative** | 作品讲解、作者介绍、创作背景、藏品故事 | **LLM 生成** | **花钱**、需持久化、可再生但不免费、非确定 | PostgreSQL（section 化内容） |
| **大二进制** | 解说语音(TTS)、图片缩略图 | TTS 引擎 / Wikimedia | 大体积 | **Cloudflare R2 对象存储** |

**关键区分**：facts 可随意重抓；narrative 是 LLM 产物，要省着生成（懒生成 + 持久化）；大二进制绝不进数据库。

### 容量直觉
- 结构化记录：10 万件 × 几 KB ≈ 几百 MB，prod 单库可扛。staging 只放样本，永远小。
- 语音/图片：体积大但放 R2，且**按需懒生成 + 缓存**，不预先批量生成（见 §4）。

---

## 3. Schema 设计：为"演进"而生

数据架构在开发中**一定会变**（加藏品类型、加描述维度）。因为数据可重现，这类演进 = **普通的版本化开发**：① 改 schema（alembic 迁移）② 改管线 ③ 各环境重跑回填。不是高危数据迁移。

让演进无痛的设计动作：

### 3.1 通用藏品表 + type 字段
- 用单一 `museum_object` 通用表 + `type` 字段（`painting` / `sculpture` / `photograph` / …）。
- **加新藏品类型 = 数据里多一种 type 值 + 放宽 Wikidata 查询，不新建表。**

### 3.2 Section 化内容模型（已具备）
- step1 schema 已是 section 化的：`SectionType` / `CategorySection` / `ObjectContentSection` + content-by-tab 端点。
- **加新描述维度（作者介绍/创作背景/藏品故事）= 新增 SectionType + 生成对应内容，不动 schema。**
- 这个 EAV/分区模式正是为内容演进设计的。App 对缺失 section 优雅降级（没有该 tab 就不显示）。

### 3.3 存原始 Wikidata JSON 整包（JSONB 列）⭐
- 抓取时把每件藏品的**完整 Wikidata 响应**原样存一个 JSONB 列，再投影出当前需要的结构化字段。
- 好处：以后想要一个当初没抽的字段，**从已存 JSON 回填即可，不必重抓全网**。原则：**抓一次、投影多次。**

### 3.4 幂等 upsert（按稳定 QID）
- 导入按 Wikidata QID 做 upsert：重跑只更新变化/新增的，随时安全重跑。
- 这也是"staging 样本可随意 drop+重建"的底气（2026-06-15 重置 staging 库即此性质）。

### 3.5 演进纪律
- **优先 additive 变更**（新增可空列、新 SectionType、新表），避免重命名/删除。
- 即便破坏性变更，因数据可重现也能恢复；但 additive 让 prod 始终稳定。

---

## 4. 大二进制：R2 + 懒生成，谁都不预灌

- **解说语音不预先批量生成**。用户第一次点某件藏品的某段讲解才生成 TTS，存进 R2 缓存（已有 TTS 缓存 + 每日预算熔断雏形）。
- R2 缓存可**跨环境共享**：同一件藏品、同一语言、同一段文本的语音是一样的。
- 成本随实际使用走，而非随藏品总量线性膨胀。
- 图片用 Wikimedia 直链 + 运行时缩略图参数，本地不存原图。

**结论：两个环境都不维护庞大的语音/图片集。**

---

## 5. 演进示例：两个场景如何走流程

### 场景 A：奥赛馆加入雕塑（新藏品类型）
1. PR：（若需）`type` 列迁移 + 放宽 Wikidata 查询（绘画 `Q3305213` → 也含雕塑 `Q860861` 等）。
2. staging：重跑 Orsay 样本导入（现含雕塑）→ 验证 UI/识别/讲解处理新类型。
3. 合并 → prod：跑导入（幂等，补进雕塑）。
4. **零数据拷贝。**

### 场景 B：绘画下加"作者介绍/创作背景/藏品故事"（新描述维度）
1. PR：新增对应 `SectionType` + 内容生成 prompt + UI tab。
2. staging：在样本上生成、验证。
3. 合并 → prod：这些**叙事 section 懒生成**（用户点到才生成那一件那一段），不批量预灌十万件。
4. **零 schema 改动、零数据拷贝。**

## 6. 没有全量数据，如何"充分测试"？拆成三种

| 测试目标 | 怎么做 | 需要全量数据吗 |
|---|---|---|
| **正确性**（代码/管线/UI/识别/讲解链路对不对） | staging 小样本，但样本须**覆盖所有 type、缺字段边界、多语言** | 否 |
| **规模/性能**（10 万行扛不扛得住） | 单独压测，用**合成数据**灌满 | 否（合成即可） |
| **上线前真数据校验** | 富化管线跑完出**抽样报告**人工审 + **金丝雀**：先富化 1 个新馆→验证→再批量 | 否（抽样 + 金丝雀） |

逻辑正确性不靠数据量；规模问题用合成数据；真数据正确性用报告 + 金丝雀。

## 7. staging 样本集定义（待细化）

样本应覆盖：
- 每种藏品 `type` 至少若干件；
- 字段完整度的边界：facts 齐全的、缺作者的、缺年代的、缺图片的、缺多语言标签的；
- 多语言（zh/en 至少）；
- 每种 `SectionType`（已生成的 / 未生成待懒加载的）；
- 识别用：一小组真实参观者照片（已有 v3 评测集可复用）。

实现：一个可重跑的 `seed_staging_sample` 脚本（固定 QID 清单或带 `--limit` 的过滤查询），让 staging 库随时可重建。

## 8. 数据流总览

```
Wikidata ──抓取──▶ museum_packs/<slug>.json (含原始 JSONB)
                          │
              幂等 upsert(按 QID)
                          ▼
        ┌─────────────────────────────────┐
        │  PostgreSQL: museum_object(+type)│
        │  + ObjectContentSection(section化)│
        └─────────────────────────────────┘
              │facts 即时               │narrative/TTS 懒生成
              ▼                          ▼
           App 读取  ◀──────────  LLM/TTS ──▶ R2 缓存(跨环境共享)

环境：同一套管线，staging 跑样本 / prod 跑全量，互不拷贝数据。
```

## 9. 待确认/后续细化

3. 富化管线的工程化：现有 `build_museum_pack.py` + `migrate_pack_to_db.py` → 是否拆成「抓取 / 投影入库 / 懒生成」三段，加抽样报告与金丝雀。
4. `seed_staging_sample` 脚本的样本清单标准。
5. R2 语音缓存的 key 规约（藏品QID + 语言 + 文本hash）与跨环境共享策略。
6. 多语言策略（Wikidata 标签语言覆盖 + LLM 叙事的语言生成）。
