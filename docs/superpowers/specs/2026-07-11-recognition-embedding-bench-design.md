# 识别引擎离线评测原型设计（DINOv2 vs CLIP 向量检索 benchmark）

> 2026-07-11 brainstorm 定稿。识别机制演进的第一步：用数据验证"向量检索能否解决识别不准"，
> 再决定是否接入线上引擎位。纯离线，不碰线上识别流程。
> 背景 spec：docs/superpowers/specs/2026-07-03-recognition-design.md（P1 GPT+OCR 闭环，已上线）。

## 背景与结论摘要

- **现状问题**：P1 链路 = GPT-4o-mini 看图猜名字 → 名字模糊匹配目录。命脉在"GPT 认不认识作品名"，
  冷门件 GPT 报错名/瞎猜，候选也不准。换更大 GPT 只抬上限，不解决本质。
- **方案**：改为按外观做实例匹配——参考图向量化建索引，用户照片向量化做余弦近邻。
  识别范围从"全世界"收缩为"库里有参考图的件"，与名气无关。
- **模型选型（本评测要裁决的）**：
  - **DINOv2 ViT-S/14（主选）**：纯视觉自监督，训练目标（同图不同增广映射到一起）与我们任务
    （游客照 = 官方图的现实增广）直接对齐；实例检索基准上小模型即超大号 CLIP；21M 参数/384 维，
    Apache 2.0。
  - **CLIP ViT-B/32 图像塔（对照）**：原 P2 计划选项。图文对齐目标导致语义相近的不同作品
    （满墙莫奈）在空间里挤在一起，实例检索是结构性短板。买一送一的文搜图不在关键路径
    （元数据文字匹配已覆盖搜索需求）。
  - DINOv3 检索更强但许可非 Apache 2.0，暂不用；引擎接口不变，以后可换。
- **已定的后续方向（不在本轮）**：识别端点做成全局为主、馆为可选提示（导航栏识别按钮直达相机，
  流程里没有"先选馆"；命中对象自带归属馆 slug 供跳详情页）；馆藏编号查询放搜索功能
  （复用 matcher.py + inventory_number 精确匹配），识别未收录卡片链过去。

## 目标（三个要回答的问题）

1. 向量检索在我们真实目录上的准确率够不够（尤其非名作）；
2. DINOv2-S 与 CLIP ViT-B/32 谁赢；
3. 三档分流阈值（直接确认 / 出候选 / 拒识）定在哪——Top-1 分数与 Top1-Top2 间距的判据校准。

## 组件

**① Embedding 引擎**（将来直接搬进线上引擎位）
- 接口：`embed(image_bytes) -> np.ndarray`，onnxruntime CPU 推理。
- 两个模型各导出 ONNX：`dinov2-vits14`（384 维）、`clip-vitb32` 图像塔（512 维）。
- 向量产物一律带模型名版本化。
- 新依赖（onnxruntime 等）放 poetry 独立 group（评测期不进生产镜像；接线上时再挪）。

**② 索引构建**
- 范围：orsay 全目录所有有参考图的件（索引按真实规模建，检索难度才真实）。
- 每件全部 ObjectImage 入索引（多视角，对雕塑关键），查询取该件所有参考图的最大相似度。
- 产物：`.npz`（向量 + qid + image_key + 模型名）。

**③ 测试集构建**（一次性脚本 + 落盘清单，评测可重复跑）
- **分层抽样**：按 `museum_object.popularity` 分层（名作 = 馆内 popularity 排名前 200，
  其余为非名作），非名作 ~70% / 名作 ~30%，共 ~100 件。
  判定线以非名作桶为准（引擎存在的意义就是救冷门件，不许靠名作拉平均）。
- **真实游客照**：Wikidata qid → Commons 分类拉同一作品的其他照片（天然角度/光照差异）。
  名作多、冷门件可能为零——有多少算多少，分桶统计。
- **合成畸变**：官方图做透视变形、反光叠加、裁切、模糊、暗光（PIL），每件数张。
  非名作桶预期以合成为主（这也预演了真实情况：冷门件只有官方图）。
- **库外干扰组**：非 orsay 目录作品图 ~30-50 件，测误接受（决定拒识阈值）。
- 平面 / 雕塑（3D）分桶标注——3D 预期偏弱，单独看。

**④ 评测脚本**
- 每张测试照 → embed → 对全索引余弦 → 汇总：
  - Top-1 / Top-3 准确率；
  - 库外误接受率（随阈值扫曲线）；
  - Top1-Top2 间距分布（三档分流判据的依据）;
  - 延迟 p50 / p95（VPS 同级 CPU 环境）。
- 报告按 模型 × 照片类型（真实/合成）× 名作/非名作 × 平面/3D 分桶，输出 markdown。

**⑤ 判定标准**（过线 → 下一轮接引擎位；不过 → 按渐进路线加码：更大模型 / 候选精排）
- 同一组阈值下同时满足：Top-1 ≥ 85%、Top-3 ≥ 95%、库外误接受 ≤ 5%；
- 且以上必须在**非名作桶单独达标**。

## 位置与产物

- 代码：`backend/scripts/recognition_bench/`。
- 产物（向量 .npz、测试集图片与清单、报告）落盘不进 git（.gitignore 覆盖）。

## 不做（YAGNI）

- 不动 API / 前端 / 线上识别流程（无需 feature flag——离线脚本天然隔离）；
- 不上 FAISS / pgvector（2k 量级 numpy 精确搜索毫秒级）；
- 不做全局识别端点与搜索功能（分别是后续两轮）；
- 不做候选精排 / OCR 增强（benchmark 失败案例说了算）。

## 后续路线（本 spec 之外，此处只记方向）

1. 评测过线 → 接引擎位：向量检索为主引擎，GPT+OCR 降级为兜底（无参考图/低分件），
   两路仍汇到同一接地匹配层，响应契约不变（match/candidates/unrecognized 三档），前端零改动；
   同轮做全局端点 `POST /api/v1/recognize?museum=<slug 可选>`（加法，老端点保留）。
2. 搜索功能（标题/作者/编号，复用 matcher.py），识别未收录卡片加入口。
